# Описание задачи

## Что решается

Задача — предсказать, будет ли страховая выплата по автомобильному страховому договору. Это бинарная классификация:

- `1` — выплата была (клиент подал страховой случай и получил деньги)
- `0` — выплаты не было

---

## Данные

**Источник:** Ethiopian Insurance Corporation — реальные данные по автострахованию.

**Файлы:**
- `data/source/motor_data11-14lats.csv` — полисы 2011–2014 гг.
- `data/source/motor_data14-2018.csv` — полисы 2014–2018 гг.

**Признаки:**

| Колонка | Что означает |
|---------|-------------|
| `SEX` | Пол страхователя |
| `INSR_BEGIN`, `INSR_END` | Начало и конец действия полиса |
| `EFFECTIVE_YR` | Год вступления в силу |
| `INSR_TYPE` | Тип страхования |
| `INSURED_VALUE` | Страховая стоимость автомобиля |
| `PREMIUM` | Страховая премия (взнос клиента) |
| `OBJECT_ID` | ID объекта (служебный) |
| `PROD_YEAR` | Год выпуска ТС |
| `SEATS_NUM` | Число мест |
| `CARRYING_CAPACITY` | Грузоподъёмность |
| `TYPE_VEHICLE` | Тип транспортного средства |
| `CCM_TON` | Объём двигателя / тоннаж |
| `MAKE` | Марка ТС |
| `USAGE` | Назначение ТС |
| `CLAIM_PAID` | Сумма выплаты (из неё создаётся целевая переменная) |

**Целевая переменная** `HAS_CLAIM` формируется автоматически:
```
HAS_CLAIM = 1, если CLAIM_PAID > 0 и не пустое
HAS_CLAIM = 0, в остальных случаях
```

Данные сортируются по дате начала полиса (`INSR_BEGIN`) — это важно для корректного батчевого обучения во времени.

---

## Поток обработки (pipeline)

Система работает батчами по **100 000 строк**. Каждый запуск `update` обрабатывает один следующий батч.

```
Данные → Батч → Анализ → Очистка → Препроцессинг → Обучение → Валидация → Продовая модель
```

### Этап 1 — Сбор данных (`data_collector.py`)

- Читает CSV-файлы, объединяет их, сортирует по времени
- Нарезает на батчи, сохраняет текущую позицию в `data/raw/state.json`
- Батч сохраняется в `data/raw/batch_XXXX.csv`

### Этап 2 — Анализ данных (`data_analyzer.py`)

**Проверка качества (Data Quality):**
- Считает процент пропусков (общий и по каждой колонке)
- Считает дубликаты
- Сохраняет отчёт в `reports/quality_batch_XXXX.json`

**Поиск ассоциативных правил** (библиотека `mlxtend`):
- Анализирует категориальные признаки (`SEX`, `INSR_TYPE`, `TYPE_VEHICLE` и др.)
- Ищет частые комбинации (min_support=0.05) и правила (confidence ≥ 0.5, lift > 1.0)
- Сохраняет топ-10 правил в `reports/rules_batch_XXXX.csv`

**Очистка:**
- Удаляет дубликаты
- Удаляет колонки с >70% пропусков (кроме `CLAIM_PAID`)
- Удаляет строки с отрицательными значениями `PREMIUM`, `INSURED_VALUE`

### Этап 3 — Подготовка данных (`data_preparator.py`)

**Разбивка:** 80% train / 20% test (хронологически — без перемешивания)

**Обработка признаков:**
- **Числовые колонки** (`PREMIUM`, `INSURED_VALUE`, `PROD_YEAR` и др.):
  - Заполнение пропусков медианой (`SimpleImputer`)
  - Масштабирование устойчивым скейлером (`RobustScaler`) — не чувствителен к выбросам
- **Категориальные колонки** (≤20 уникальных значений):
  - Заполнение пропусков модой
  - One-Hot Encoding (`OneHotEncoder`, неизвестные значения игнорируются)
- Служебные колонки (`INSR_BEGIN`, `INSR_END`, `OBJECT_ID`, `CLAIM_PAID`) удаляются

**Артефакт:** препроцессор сохраняется в `models/preprocessor_batch_XXXX.pkl`. При последующих батчах переиспользуется существующий препроцессор (без повторного fit).

### Этап 4 — Обучение (`trainer.py`)

Обучаются две модели:

**Decision Tree (`DecisionTreeClassifier`):**
- `max_depth=8` — ограничивает сложность, снижает переобучение
- `min_samples_leaf=30` — в листе должно быть достаточно примеров
- `class_weight="balanced"` — компенсирует дисбаланс классов (страховые случаи редки)
- Обучается заново на каждом батче

**MLP (Многослойный перцептрон, `MLPClassifier`):**
- Архитектура: два скрытых слоя — 64 и 32 нейрона
- `max_iter=300` при обучении с нуля
- Начиная со второго батча — дообучается методом `partial_fit` (инкрементальное обучение)
- Это позволяет сети "помнить" знания предыдущих батчей

### Этап 5 — Валидация (`validator.py`)

Метрики на тестовой выборке:

| Метрика | Что показывает |
|---------|---------------|
| `accuracy` | Доля правильных предсказаний |
| `f1_weighted` | F1 с учётом дисбаланса классов |
| `f1_claim` | F1 именно для класса "выплата" (ключевая метрика) |
| `ROC-AUC` | Качество ранжирования (у моделей с predict_proba) |

Лучшая модель выбирается по `f1_claim` — это наиболее важная метрика, так как нас интересует точность предсказания страховых случаев.

История всех батчей сохраняется в `models/metrics_history.json`.

### Этап 6 — Обслуживание (`server.py`)

- Лучшая модель и препроцессор сохраняются вместе в `models/production_model.pkl`
- Функция `predict()` загружает пакет, применяет препроцессор и возвращает предсказания
- Каждый вызов логируется в `logs/performance.jsonl` (дата, кол-во строк, время в мс)

## Сохранение артефактов

| Артефакт | Путь | Описание |
|----------|------|----------|
| Батч данных | `data/raw/batch_XXXX.csv` | Сырые данные батча |
| Состояние | `data/raw/state.json` | Номер следующего батча |
| Отчёт качества | `reports/quality_batch_XXXX.json` | Пропуски, дубликаты |
| Правила | `reports/rules_batch_XXXX.csv` | Ассоциативные правила |
| Препроцессор | `models/preprocessor_batch_XXXX.pkl` | Fit-трансформер |
| Модель батча | `models/tree_batch_XXXX.pkl`, `mlp_batch_XXXX.pkl` | Обученные модели |
| Продовая модель | `models/production_model.pkl` | Лучшая модель + препроцессор |
| История метрик | `models/metrics_history.json` | Все батчи, все метрики |
| Лог инференса | `logs/performance.jsonl` | Скорость работы |

## CI/CD подход

Используется **базовый сценарий** GitHub Actions: при push/pull request запускается один батч обучения (`python run.py -mode update`). Логи сохраняются как артефакт GitHub Actions.
